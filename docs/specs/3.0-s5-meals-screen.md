## Context for Claude Code

This is session 5 of 8. The CRUD pattern is established from S3 and S4. This screen has more specialized UI than the previous two — the weekly meal plan view isn't a standard card list, and recipes have structured sub-data (ingredients, instructions).

**Data scoping:** `recipes`, `meal_plans`, and `shopping_lists` all have `user_id` and `household_id`, scoped by RLS on `household_id`. Both users see the same meal plans and recipes.

**Key data model insight:** The `create_meal_plan` MCP tool takes the entire week at once — an array of `{ day_of_week, meal_type, recipe_id?, custom_meal? }` entries plus a `week_start` date. This means meal plans are created/updated at the week level, not per individual meal. The UI should reflect this: editing a single meal slot updates (or creates) the full week plan.

## This week tab

The primary view — a compact weekly meal plan.

### WeekNav

Navigation bar at the top of the tab:
- Left arrow: previous week
- Center: "This week" / "Week of Apr 7" (formatted label)
- Right arrow: next week
- Tapping the center label returns to the current week

### WeekView

A vertical list of days (Monday through Sunday), each showing its planned meals.

```
┌─────────────────────────────────┐
│ ← Week of Mar 30               →│
├─────────────────────────────────┤
│ MONDAY, MAR 30                   │
│ ┌─────────────────────────────┐ │
│ │ Dinner: Chicken tikka masala│ │
│ └─────────────────────────────┘ │
│                                  │
│ TUESDAY, MAR 31                  │
│ ┌─────────────────────────────┐ │
│ │ Dinner: Sheet pan chicken   │ │
│ └─────────────────────────────┘ │
│                                  │
│ WEDNESDAY, APR 1                 │
│ (no meals planned)        [+ Add]│
│                                  │
│ ...                              │
└─────────────────────────────────┘
```

### DayColumn

A single day in the week view.

- **Header:** Day name + date (SectionHeader style: "Monday, Mar 30")
- **Meal slots:** One card per planned meal for that day
- **Empty state:** If no meals for that day, show muted "No meals planned" + an "Add" link
- **Add button:** Small "+" button or "Add meal" link at the bottom of each day

### MealSlot

A single meal entry within a day. Rendered as a compact card.

```tsx
type MealSlotProps = {
  dayOfWeek: string
  mealType: string              // 'breakfast' | 'lunch' | 'dinner' | 'snack'
  recipeName?: string           // from linked recipe
  customMeal?: string           // freetext if no recipe linked
  servings?: number
  notes?: string
  onTap: () => void             // opens edit/detail
}
```

**Display:**
- Left accent line colored by meal type (breakfast = amber, lunch = blue, dinner = teal, snack = gray)
- Meal type label in small muted text: "Dinner"
- Meal name: recipe name OR custom_meal text
- If linked to a recipe: small "View recipe →" link
- Tapping opens MealSlotForm for editing

### MealSlotForm

Modal or DetailPanel for adding/editing a single meal slot.

**Fields:**
- Day (pre-filled, read-only — from which day was tapped)
- Meal type (select: Breakfast, Lunch, Dinner, Snack)
- **Recipe OR freetext** — this is the key UX decision:
  - Toggle or tab: "Pick a recipe" / "Custom meal"
  - **Pick a recipe:** Opens RecipePicker (search existing recipes, tap to select). Shows selected recipe name.
  - **Custom meal:** Free text input (placeholder: "e.g., Takeout pizza, Leftovers")
- Servings (number, optional)
- Notes (text, optional, placeholder: "e.g., Make extra for lunches")

**Save behavior:**
- Saving a meal slot updates the entire week's meal plan. The frontend should:
  1. Fetch the current week's meal plan
  2. Add/update the meal entry in the `meals` array
  3. Call `create_meal_plan` (which creates or replaces the week plan) with the full updated array
- This is an upsert pattern at the week level, not a per-meal insert

**Delete a meal slot:** Remove the entry from the `meals` array and re-save the week plan.

### RecipePicker

A search-and-select component used within MealSlotForm.

```tsx
type RecipePickerProps = {
  onSelect: (recipe: { id: string; name: string }) => void
  onCancel: () => void
}
```

**Behavior:**
- Search input at the top
- Debounced search against recipes (by name, cuisine, tag, ingredient)
- Results as a simple tappable list (recipe name + cuisine + tags)
- Tapping a result calls `onSelect` with the recipe's id and name
- "Cancel" to go back to freetext mode

## Recipes tab

A browsable collection of saved recipes.

### List view

- **Search bar** at the top
- **Filter pills** for cuisine types (derived from data) and/or tags
- Recipes as cards, sorted alphabetically

### RecipeCard

```tsx
<Card
  title={recipe.name}
  subtitle={buildSubtitle(recipe)}         // "Italian · 45 min · ★ 4"
  badges={recipe.tags?.map(t => ({ label: t, variant: 'gray' }))}
  onTap={() => openDetail(recipe)}
/>
```

**Subtitle logic:** Combine available metadata:
- Cuisine (if set)
- Total time: prep + cook minutes, formatted as "45 min" or "1h 15m"
- Rating: "★ 4" (if set)
- Separated by " · "

### RecipeDetail

Full recipe view in DetailPanel. This is the most content-rich detail view in the app.

**Sections:**
1. **Name** — recipe title, prominent
2. **Meta row** — cuisine, total time, servings, rating — rendered as a compact horizontal row of small metric boxes (similar to StatRow but inline)
3. **Tags** — horizontal badge row
4. **Ingredients** — list of ingredients, each formatted as "2 cups flour" (quantity + unit + name). Clean, readable list with subtle left border or bullet.
5. **Instructions** — numbered steps, each as a paragraph. Clear step numbers.
6. **Notes** — if set, rendered below instructions in muted style

**Header buttons:** Edit → RecipeForm
**Footer (owner only):** Delete → ConfirmDialog

**"Add to meal plan" button:** Prominent button at the bottom of the detail view (above delete). Tapping opens a quick dialog: pick a day + meal type for the current week, and the recipe gets added to the plan. This is a shortcut so users can browse recipes and plan meals from the recipe detail.

### RecipeForm

The most complex form in the app, due to ingredients and instructions being dynamic lists.

**Fields:**
- Name (text, required)
- Cuisine (text, optional, placeholder: "e.g., Italian, Mexican, Thai")
- Servings (number, optional)
- Prep time (number, optional, label: "Prep time (minutes)")
- Cook time (number, optional, label: "Cook time (minutes)")
- Rating (select: 1–5, optional)
- Tags (text, comma-separated, optional)
- Notes (textarea, optional)

**Ingredients section (IngredientList):**
- Dynamic list of ingredient rows
- Each row: three inputs side by side — quantity (text, narrow), unit (text or select, narrow), name (text, wide)
- "Add ingredient" button at the bottom
- Remove button (✕) on each row
- At least one ingredient required

```
Ingredients
┌───────┐ ┌───────┐ ┌────────────────┐  ✕
│ 2     │ │ cups  │ │ flour          │
└───────┘ └───────┘ └────────────────┘
┌───────┐ ┌───────┐ ┌────────────────┐  ✕
│ 1     │ │ tsp   │ │ salt           │
└───────┘ └───────┘ └────────────────┘
              [+ Add ingredient]
```

**Responsive note:** On a 375px screen, three inputs side by side is tight. Options:
- Stack quantity + unit on one line, name on the next
- Or use a single text input per ingredient ("2 cups flour") and parse later
- Recommend: single text input per ingredient for v1. Simpler to build, easier to type on mobile. Store as `{ name: "flour", quantity: "2", unit: "cups" }` by parsing, or store the whole string and let the MCP tool parse it. Check how `add_recipe` expects the data.

**Actually — the MCP tool expects structured `{ name, quantity, unit }` objects.** So the form needs to capture them separately. Use two rows per ingredient on mobile:
```
┌────────────────────────────────┐  ✕
│ flour                          │
│ 2 cups                         │
└────────────────────────────────┘
```
Or three inputs in a row if they fit. Claude Code should test both layouts at 375px and pick whichever works.

**Instructions section (InstructionList):**
- Dynamic list of numbered steps
- Each step: a textarea (or tall text input), numbered automatically (1, 2, 3...)
- "Add step" button at the bottom
- Remove button on each step
- At least one step required
- Steps auto-renumber when one is removed

```
Instructions
1. ┌────────────────────────────┐  ✕
   │ Preheat oven to 375°F     │
   └────────────────────────────┘
2. ┌────────────────────────────┐  ✕
   │ Mix dry ingredients        │
   └────────────────────────────┘
            [+ Add step]
```

**Save handler:**
- Collect all form fields
- Build the `ingredients` array as `[{ name, quantity, unit }, ...]`
- Build the `instructions` array as `[string, string, ...]`
- Create: `supabase.from('recipes').insert({ ... })`
- Update: `supabase.from('recipes').update({ ... }).eq('id', recipe.id)`

## Shopping tab

### Overview

Shows the generated shopping list for the current week's meal plan. The core flow:
1. User plans meals on the "This week" tab
2. User switches to "Shopping" tab
3. Taps "Generate shopping list" (or it auto-generates)
4. Sees an aggregated ingredient list grouped by category or alphabetically
5. Can check off items while shopping

### ShoppingList

**Top section:**
- Week label: "Shopping list for week of Mar 30"
- "Generate" / "Refresh" button — calls `generate_shopping_list` with the current week's `week_start`
- **"Copy for Instacart" button** — appears when unpurchased items exist; copies only item names (no quantities, no units) to clipboard via `navigator.clipboard.writeText()`, one item per line; shows "Copied!" for 1.5s then reverts. Instacart's text input doesn't accept quantities, so only `item.name` is copied.

**List:**
- Items grouped logically (if the backend groups them) or displayed as a flat alphabetical list
- Each item: checkbox + ingredient text (e.g., "2 cups flour", "1 lb chicken thighs")
- Tapping the checkbox strikes through the item (client-side state — check if the backend supports persisting checked state)
- Checked items move to the bottom of the list or stay in place with strikethrough

**Empty state:** "No meal plan for this week. Plan some meals first!" with a link/button to the "This week" tab.

**Implementation note:** The `generate_shopping_list` MCP tool takes `week_start` and presumably returns the aggregated list. Check what it returns — it might save to the `shopping_lists` table, or it might return the list directly. The frontend should handle both:
- If it saves to `shopping_lists`: fetch the list after generating
- If it returns directly: render the response and optionally save it

**Check-off persistence:** Check whether `shopping_lists` has a way to store checked state (e.g., `items` JSONB with a `checked` boolean per item). If not, checked state is session-only (resets on page refresh). That's acceptable for v1 — note it as a future improvement.

## UX flows worth getting right

### Adding a meal to the plan

Two entry points:
1. **From the week view:** Tap "+" on a day → MealSlotForm opens → pick recipe or type custom meal → save
2. **From recipe detail:** Tap "Add to meal plan" → quick dialog (pick day + meal type) → save

Both flows ultimately call `saveMealPlan` with the updated meals array.

### Editing a planned meal

Tap the meal slot → MealSlotForm opens prepopulated → change recipe or meal type → save (re-saves the full week plan)

### Removing a planned meal

In MealSlotForm, add a "Remove meal" button (red text, at bottom). This removes the entry from the `meals` array and re-saves the week plan. No ConfirmDialog needed — it's easy to re-add.

### Planning a new week

Navigate to a future week → empty state: "No meals planned yet" → start adding via day "+" buttons. The first save creates the `meal_plans` row for that week.

## Edge cases

1. **No recipes exist:** Recipes tab shows EmptyState: "No recipes saved. Tap + to add one." Meal planning still works via custom meals.
2. **No meal plan for current week:** Week view shows "No meals planned" under each day with "+" links.
3. **Recipe deleted that's linked to a meal plan:** The meal slot should still show the recipe name (denormalized in the meals JSONB) even if the recipe is deleted. Just hide the "View recipe →" link. Check if recipe name is stored in the meal plan entry or only recipe_id.
4. **Shopping list with no meal plan:** Show "No meal plan for this week" instead of generating an empty list.
5. **Very long recipe (20+ ingredients, 15+ steps):** DetailPanel should scroll. Ingredient and instruction lists should have no max height.
6. **Duplicate ingredients across recipes:** Shopping list generation should aggregate (e.g., two recipes each needing 1 cup flour = 2 cups flour). This is presumably handled by `generate_shopping_list` on the backend.
7. **Meal plan for a past week:** Allow viewing but maybe add a muted "Past week" indicator. Don't prevent editing — someone might want to update a past plan for reference.
8. **Custom meal in week view then later creating a matching recipe:** No auto-linking needed in v1. The custom_meal stays as text.
