## Context for Claude Code

This is session 5 of 8. The CRUD pattern is established from S3 and S4. This screen has more specialized UI than the previous two вҖ” the weekly meal plan view isn't a standard card list, and recipes have structured sub-data (ingredients, instructions).

**Data scoping:** `recipes`, `meal_plans`, and `shopping_lists` all have `user_id` and `household_id`, scoped by RLS on `household_id`. Both users see the same meal plans and recipes.

**Key data model insight:** The `create_meal_plan` MCP tool takes the entire week at once вҖ” an array of `{ day_of_week, meal_type, recipe_id?, custom_meal? }` entries plus a `week_start` date. This means meal plans are created/updated at the week level, not per individual meal. The UI should reflect this: editing a single meal slot updates (or creates) the full week plan.

### DayColumn

A single day in the week view.

- **Header:** Day name + date (SectionHeader style: "Monday, Mar 30")
- **Meal slots:** One card per planned meal for that day
- **Empty state:** If no meals for that day, show muted "No meals planned" + an "Add" link
- **Add button:** Small "+" button or "Add meal" link at the bottom of each day

### MealSlot

A single meal entry within a day. Rendered as a compact card.

```tsx
type MealSlotProps = {
  dayOfWeek: string
  mealType: string              // 'breakfast' | 'lunch' | 'dinner' | 'snack'
  recipeName?: string           // from linked recipe
  customMeal?: string           // freetext if no recipe linked
  servings?: number
  notes?: string
  onTap: () => void             // opens edit/detail
}
```

**Display:**
- Left accent line colored by meal type (breakfast = amber, lunch = blue, dinner = teal, snack = gray)
- Meal type label in small muted text: "Dinner"
- Meal name: recipe name OR custom_meal text
- If linked to a recipe: small "View recipe вҶ’" link
- Tapping opens MealSlotForm for editing

### MealSlotForm

Modal or DetailPanel for adding/editing a single meal slot.

**Fields:**
- Day (pre-filled, read-only вҖ” from which day was tapped)
- Meal type (select: Breakfast, Lunch, Dinner, Snack)
- **Recipe OR freetext** вҖ” this is the key UX decision:
  - Toggle or tab: "Pick a recipe" / "Custom meal"
  - **Pick a recipe:** Opens RecipePicker (search existing recipes, tap to select). Shows selected recipe name.
  - **Custom meal:** Free text input (placeholder: "e.g., Takeout pizza, Leftovers")
- Servings (number, optional)
- Notes (text, optional, placeholder: "e.g., Make extra for lunches")

**Save behavior:**
- Saving a meal slot updates the entire week's meal plan. The frontend should:
  1. Fetch the current week's meal plan
  2. Add/update the meal entry in the `meals` array
  3. Call `create_meal_plan` (which creates or replaces the week plan) with the full updated array
- This is an upsert pattern at the week level, not a per-meal insert

**Delete a meal slot:** Remove the entry from the `meals` array and re-save the week plan.

### RecipePicker

A search-and-select component used within MealSlotForm.

```tsx
type RecipePickerProps = {
  onSelect: (recipe: { id: string; name: string }) => void
  onCancel: () => void
}
```

**Behavior:**
- Search input at the top
- Debounced search against recipes (by name, cuisine, tag, ingredient)
- Results as a simple tappable list (recipe name + cuisine + tags)
- Tapping a result calls `onSelect` with the recipe's id and name
- "Cancel" to go back to freetext mode

## Shopping tab

### Overview

Shows the generated shopping list for the current week's meal plan. The core flow:
1. User plans meals on the "This week" tab
2. User switches to "Shopping" tab
3. Taps "Generate shopping list" (or it auto-generates)
4. Sees an aggregated ingredient list grouped by category or alphabetically
5. Can check off items while shopping

### ShoppingList

**Top section:**
- Week label: "Shopping list for week of Mar 30"
- "Generate" / "Refresh" button вҖ” calls `generate_shopping_list` with the current week's `week_start`

**List:**
- Items grouped logically (if the backend groups them) or displayed as a flat alphabetical list
- Each item: checkbox + ingredient text (e.g., "2 cups flour", "1 lb chicken thighs")
- Tapping the checkbox strikes through the item (client-side state вҖ” check if the backend supports persisting checked state)
- Checked items move to the bottom of the list or stay in place with strikethrough

**Empty state:** "No meal plan for this week. Plan some meals first!" with a link/button to the "This week" tab.

**Implementation note:** The `generate_shopping_list` MCP tool takes `week_start` and presumably returns the aggregated list. Check what it returns вҖ” it might save to the `shopping_lists` table, or it might return the list directly. The frontend should handle both:
- If it saves to `shopping_lists`: fetch the list after generating
- If it returns directly: render the response and optionally save it

**Check-off persistence:** Check whether `shopping_lists` has a way to store checked state (e.g., `items` JSONB with a `checked` boolean per item). If not, checked state is session-only (resets on page refresh). That's acceptable for v1 вҖ” note it as a future improvement.

## UX flows worth getting right

### Adding a meal to the plan

Two entry points:
1. **From the week view:** Tap "+" on a day вҶ’ MealSlotForm opens вҶ’ pick recipe or type custom meal вҶ’ save
2. **From recipe detail:** Tap "Add to meal plan" вҶ’ quick dialog (pick day + meal type) вҶ’ save

Both flows ultimately call `saveMealPlan` with the updated meals array.

### Editing a planned meal

Tap the meal slot вҶ’ MealSlotForm opens prepopulated вҶ’ change recipe or meal type вҶ’ save (re-saves the full week plan)

### Removing a planned meal

In MealSlotForm, add a "Remove meal" button (red text, at bottom). This removes the entry from the `meals` array and re-saves the week plan. No ConfirmDialog needed вҖ” it's easy to re-add.

### Planning a new week

Navigate to a future week вҶ’ empty state: "No meals planned yet" вҶ’ start adding via day "+" buttons. The first save creates the `meal_plans` row for that week.

## Edge cases

1. **No recipes exist:** Recipes tab shows EmptyState: "No recipes saved. Tap + to add one." Meal planning still works via custom meals.
2. **No meal plan for current week:** Week view shows "No meals planned" under each day with "+" links.
3. **Recipe deleted that's linked to a meal plan:** The meal slot should still show the recipe name (denormalized in the meals JSONB) even if the recipe is deleted. Just hide the "View recipe вҶ’" link. Check if recipe name is stored in the meal plan entry or only recipe_id.
4. **Shopping list with no meal plan:** Show "No meal plan for this week" instead of generating an empty list.
5. **Very long recipe (20+ ingredients, 15+ steps):** DetailPanel should scroll. Ingredient and instruction lists should have no max height.
6. **Duplicate ingredients across recipes:** Shopping list generation should aggregate (e.g., two recipes each needing 1 cup flour = 2 cups flour). This is presumably handled by `generate_shopping_list` on the backend.
7. **Meal plan for a past week:** Allow viewing but maybe add a muted "Past week" indicator. Don't prevent editing вҖ” someone might want to update a past plan for reference.
8. **Custom meal in week view then later creating a matching recipe:** No auto-linking needed in v1. The custom_meal stays as text.
