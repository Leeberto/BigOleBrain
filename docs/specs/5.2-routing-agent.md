# Feature 5.2: Routing Agent — action classification + dispatch

**Category:** `supabase/functions/routing-agent/index.ts` or task within existing task engine

## Done when

- Agent evaluates actions where `agent_status = 'pending_routing'`
- Classifies each action as `human_required` or `agent_handleable`
- For human-required actions: sets `assigned_to = 'human'`, `agent_status = 'handed_back'`
- For agent-handleable actions: looks up the capability registry, sets `assigned_to = 'agent'`, sets `agent_capability` to the matched capability, sets `agent_status = 'agent_working'`, and dispatches to the execution agent
- This test works:
  ```
  Action: "Call Integrity Air for ductwork estimate"
  → Routing agent classifies as human_required (physical phone call)
  → assigned_to: human, agent_status: handed_back

  Action: "Research current pricing for Daikin mini-split systems"
  → Routing agent classifies as agent_handleable
  → Matches capability: "web_research"
  → Dispatches to research execution agent
  ```

## Context

This is the decision-maker in the pipeline. It sits between triage (which creates actions) and execution (which does work). Its job is answering: "Can a machine handle this, or does a human need to?"

```
[Triage Agent] → Actions table → [Routing Agent] → human OR [Execution Agent]
```

### Classification heuristics

Actions that require a human:
- Physical actions (go somewhere, move something, fix something with hands)
- Phone calls and in-person conversations
- Signing documents, legal decisions
- Anything requiring judgment you haven't delegated (hiring, firing, large purchases)
- Relationship-sensitive communications (difficult conversations, negotiations)

Actions an agent can handle:
- Research and information gathering
- Drafting emails, messages, documents
- Scheduling (if calendar access exists)
- Data analysis and summarization
- Building decks, reports, formatted outputs
- Updating spreadsheets or databases
- Routine notifications and reminders

## What to build

### Capability registry table

```sql
create table agent_capabilities (
  id uuid primary key default gen_random_uuid(),
  name text not null unique,          -- 'web_research', 'email_draft', 'deck_builder', 'scheduling'
  description text not null,          -- what this capability does, used by routing LLM
  handler_type text not null,         -- 'edge_function', 'mcp_tool', 'external_api'
  handler_config jsonb not null,      -- endpoint URL, function name, required params
  required_integrations text[],       -- ['gmail', 'google_calendar'] — what must be configured
  enabled boolean default true,
  created_at timestamptz default now()
);
```

Seed with initial capabilities:
```sql
insert into agent_capabilities (name, description, handler_type, handler_config, required_integrations) values
  ('web_research', 'Search the web and summarize findings on a topic', 'edge_function', '{"function": "research-agent"}', '{}'),
  ('email_draft', 'Draft an email based on context and intent', 'edge_function', '{"function": "email-agent"}', '{gmail}'),
  ('deck_builder', 'Build a presentation deck from gathered context', 'edge_function', '{"function": "deck-agent"}', '{}'),
  ('summarize', 'Summarize information from thoughts and actions', 'edge_function', '{"function": "summarize-agent"}', '{}'),
  ('alert_notify', 'Send a notification or reminder', 'edge_function', '{"function": "notify-agent"}', '{email}');
```

### Routing logic

1. Query `actions` where `agent_status = 'pending_routing'`
2. For each action, send to LLM with this prompt structure:
   ```
   You are a task router. Given an action and a list of available agent capabilities,
   decide: does this require a human, or can an agent handle it?

   Action: "{action.content}"
   Context: {linked thought content if available}

   Available capabilities:
   - web_research: Search the web and summarize findings
   - email_draft: Draft an email based on context and intent
   - deck_builder: Build a presentation from gathered context
   - summarize: Summarize information from thoughts and actions
   - alert_notify: Send a notification or reminder

   Respond with JSON:
   {"classification": "human_required" | "agent_handleable", "capability": "name" | null, "reasoning": "one sentence"}
   ```
3. Based on LLM response:
   - `human_required` → update action: `assigned_to = 'human'`, `agent_status = 'handed_back'`
   - `agent_handleable` → update action: `assigned_to = 'agent'`, `agent_capability = capability_name`, `agent_status = 'agent_working'` → dispatch to execution agent

### Dispatch mechanism

When routing to an agent, the router:
1. Looks up the capability in `agent_capabilities`
2. Reads `handler_type` and `handler_config`
3. Invokes the handler (Edge Function call, MCP tool call, or external API)
4. Passes the action content + any linked thought context as input

### Trigger

Same options as triage agent — scheduled (every 15 minutes) or database trigger on actions insert/update. Start with scheduled.

### Fallback behavior

- If no capability matches → classify as `human_required`
- If the matched capability is `enabled = false` → classify as `human_required`
- If dispatch fails → set `agent_status = 'needs_review'`, leave for human

## Dependencies

- 5.1 (triage agent) — produces the `pending_routing` actions this agent evaluates
- 1.1a (UUID exposure) — needs action and thought IDs
- LLM API access for classification
