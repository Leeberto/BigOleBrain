# Feature 5.1: Triage Agent — thought → action auto-creation

**Category:** `supabase/functions/triage-agent/index.ts` or task within existing task engine

## Done when

- Agent scans `thoughts` table on a schedule (or trigger)
- Identifies thoughts that imply work (type = `action_item`, `question`, `task`, or LLM-assessed)
- Auto-creates rows in `actions` table with `thought_id` linked
- Skips thoughts that already have a linked action
- Tags auto-created actions with `source:triage-agent` so you can distinguish them from manual
- Adds `assigned_to` field on actions: defaults to `human` for triage-created actions (routing agent reassigns in Feature 5.2)
- This test works:
  ```
  Capture a thought: "Need to get a quote from Integrity Air for the ductwork replacement"
  → Triage agent picks it up on next scan
  → Action auto-created: "Get a quote from Integrity Air for ductwork replacement"
     linked to source thought, tagged source:triage-agent, assigned_to: human
  ```

## Context

Today you manually say "create an action from that thought." The triage agent removes that step. It watches the thoughts table and decides what implies work. This is the first stage of the agent pipeline:

```
Capture → [Triage Agent] → Actions table → [Routing Agent] → [Execution Agents]
```

## What to build

### Schema migration

```sql
-- Add assignment tracking to actions
alter table actions add column assigned_to text not null default 'human'
  check (assigned_to in ('human', 'triage-agent', 'routing-agent', 'agent'));
alter table actions add column agent_capability text;  -- which capability handles this, set by routing agent
alter table actions add column agent_status text
  check (agent_status in ('pending_triage', 'pending_routing', 'agent_working', 'needs_review', 'handed_back'));
```

### Triage logic

1. Query `thoughts` where:
   - `created_at > last_triage_run` (only scan new thoughts)
   - No existing row in `actions` where `actions.thought_id = thoughts.id`
2. For each unprocessed thought, evaluate whether it implies work:
   - **Fast path:** if `metadata->>'type'` is `action_item` or `task` → always create action
   - **LLM path:** for other types, send content to LLM with prompt: "Does this thought imply a specific action someone needs to take? If yes, state the action in one sentence. If no, respond NO."
3. For thoughts that imply work:
   - Create action with `content` = the action statement (from LLM or thought content)
   - Set `thought_id` = source thought UUID
   - Set `assigned_to = 'human'` (default until routing agent evaluates)
   - Set `agent_status = 'pending_routing'`
   - Add tag `source:triage-agent`
4. Update `last_triage_run` timestamp

### Trigger options (pick one)

- **Option A: Scheduled** — runs every 15 minutes via task engine or pg_cron. Simple, predictable.
- **Option B: Database trigger** — Postgres trigger on `thoughts` insert fires a webhook to the triage function. Real-time but more complex.
- **Recommendation:** Start with Option A. You can add Option B later for lower latency.

### Edge Function

`supabase/functions/triage-agent/index.ts` — or register as a task type in the task engine. Either way, it:
1. Queries unprocessed thoughts
2. Runs triage logic (fast path + LLM path)
3. Creates actions
4. Logs what it did

### Guard against noise

Not every thought should become an action. The LLM evaluation step is the filter. The prompt should be conservative — when in doubt, don't create an action. You can always manually promote thoughts that the triage agent skips. Better to miss some than to flood the actions table with noise.

## Dependencies

- 1.1a (UUID exposure) — needs thought IDs to link actions
- LLM API access (OpenRouter or Anthropic) for the evaluation step
