## Context for Claude Code

This is session 6 of 8. The three data screens (Actions S3, Household S4, Meals S5) are built. This session creates the composite home view that pulls summary data from each.

**This is the killer feature of the app.** It's what Lee and Liv open every morning. It needs to load fast, be scannable in 30 seconds on a phone, and surface the right things without requiring taps.

**Key principle: this screen is a read-only aggregator.** It queries data from existing tables using existing query functions from `lib/queries/`. It does NOT have its own data model, its own table, or any mutation logic. Every card links out to the screen where the real work happens.

**Reuse existing query functions.** The query files from S3–S5 (`lib/queries/actions.ts`, `lib/queries/maintenance.ts`, `lib/queries/meal-plans.ts`) already have the functions needed. Import and call them — don't duplicate queries.

## Screen layout (top to bottom)

Everything below renders in a single scrollable column. No tabs, no filters — just a vertical feed of cards.

### 1. Greeting

```tsx
// Top of page, no card wrapper — just text
"Good morning, Lee"     // before 12pm
"Good afternoon, Lee"   // 12pm–5pm
"Good evening, Lee"     // after 5pm
```

- User's first name from auth profile (or parsed from email if no display name)
- Today's date below the greeting: "Tuesday, April 1"
- Keep it simple — no weather, no quotes

### 2. BriefingStats (StatRow)

Three metric boxes in a row:

| Metric | Query | Color/variant |
|--------|-------|---------------|
| Overdue | actions where `due_date < today` and status in (open, in_progress) | Red text if > 0 |
| Due today | actions where `due_date = today` and status in (open, in_progress) | Amber text if > 0 |
| Open | total actions where status in (open, in_progress) | Neutral |

Use the existing query from `lib/queries/actions.ts`. A single fetch of active actions can compute all three counts client-side.

### 3. Today's actions (ActionsSection)

**Priority: 1 (most important section)**

Shows actions that need attention today: overdue first, then due today, then in-progress with no date.

**Data:** Active actions where `due_date <= today` OR `status = 'in_progress'`. Ordered: overdue first (oldest first), then due today, then in-progress. Limit to 5 items — if more exist, show "View all X actions →" link at the bottom.

**Card display:** Reuse ActionCard from S3 (or compose Card the same way). Each card shows content, due date badge, and tags.

**Tapping a card:** Navigate to `/actions` with the action detail open. Two options:
- Simple: just navigate to `/actions` (user finds it in the list)
- Better: navigate to `/actions?id={action.id}` and have the Actions page auto-open that action's DetailPanel

Go with whichever is simpler to implement. The simple approach is fine for v1.

**Empty state:** "Nothing due today. Nice!" — no card, just a muted line.

### 4. Agent activity (AgentSection) — Lee only

**Priority: 2**

Shows recent agent pipeline activity. **Only renders when `isOwner` is true.**

**Data source:** This depends on what agent infrastructure exists. There are three likely scenarios:

**Scenario A — Agent tables exist (Phase 5 built):**
Query the agent events/logs table for the most recent 3–5 entries. Display as cards with agent name, action taken, timestamp, and status badge.

**Scenario B — Task engine logs exist (1.1d built, Phase 5 not yet):**
Query `scheduled_task_runs` or similar for recent automated task outputs (morning briefing generation, stale loop scans, alert digests). Display as cards showing task name, result summary, and timestamp.

**Scenario C — Neither exists yet:**
Show a placeholder: "Agent pipeline coming soon" in muted text. Don't query anything. This is the most likely scenario for the first deploy.

**Implementation:** Claude Code should check the database for agent-related tables. Build Scenario A or B if the tables exist, otherwise fall back to Scenario C. Structure the component so it's easy to upgrade later — keep the agent query in its own function in a new `lib/queries/agent-events.ts` file (even if it's just a stub returning an empty array).

  // TODO: Query agent_events table once Phase 5 is built

**Card display (when data exists):**
- Agent name badge (purple): "Triage agent", "Research agent", "Routing agent"
- Action description: "Created action from thought about refinancing"
- Status badge: "New action" (purple), "Complete" (green), "Routed" (blue), "Needs review" (amber)
- Timestamp: "2h ago", "Yesterday"

**Tapping a card:** Navigate to `/agent-feed` (or `/agent-feed?id=X` if deep linking is built).

**Empty state (no recent activity):** "No recent agent activity." — muted line, no card.

### 5. Today's meals (MealsSection)

**Priority: 3**

Shows today's planned meals from the current week's meal plan.

**Data:** Fetch the current week's meal plan using `getMealPlan(currentWeekStart)` from `lib/queries/meal-plans.ts`. Filter to today's `day_of_week` entries.

**Display:** One compact card per meal, with a left accent border colored by meal type (matching the MealSlot component from S5):
- Breakfast = amber accent
- Lunch = blue accent
- Dinner = teal accent
- Snack = gray accent

Each card shows:
- Meal type label (small, muted): "Dinner"
- Meal name: recipe name or custom_meal text

**Tapping a card:** Navigate to `/meals` (the This Week tab).

**Empty state:** "No meals planned today." — muted line. Optionally: "Plan meals →" link to `/meals`.

### 6. Stale items (StaleSection)

**Priority: 4**

Surfaces things that are slipping through the cracks: old open actions with no due date, actions that have been in-progress for a long time, etc.

**Data:** Active actions where:
- `due_date is null` and `created_at < 14 days ago` (no deadline, getting old)
- OR `status = 'in_progress'` and `updated_at < 7 days ago` (stalled)

Limit to 3 items. If more exist, show "View X stale items →" link.

**Card display:** Standard Card with content and a gray "Stale" badge + age indicator ("Created 3 weeks ago").

**Tapping a card:** Navigate to `/actions`.

**Empty state:** Don't render the section at all. No need for "No stale items" — the absence is good news and doesn't need to be called out.

### 7. Upcoming maintenance (MaintenanceSection)

**Priority: 5**

Shows maintenance tasks due in the next 14 days.

**Data:** Use `lib/queries/maintenance.ts` — query maintenance_tasks where `next_due` is within the next 14 days, ordered by `next_due` ascending. Include overdue tasks. Limit to 3 items.

**Card display:** Standard Card with:
- Task name
- Urgency badge: "Overdue X days" (red), "Due in X days" (amber/green)
- Last completed date as subtitle (if available)

**Tapping a card:** Navigate to `/household` (Maintenance tab).

**Empty state:** Don't render the section at all. No upcoming maintenance = nothing to show.

## Section rendering rules

- Each section has a SectionHeader label ("Today's actions", "Agent activity", "Today's meals", "Needs attention", "Upcoming maintenance")
- Sections render in the fixed priority order listed above, regardless of which have data
- Sections with no data either show their empty state text (sections 3, 4, 5) or hide entirely (sections 6, 7)
- Agent activity (section 4) is hidden entirely for members (`!isOwner`)

## Loading behavior

The briefing fetches data from multiple tables. Don't block the entire page on all queries completing.

**Progressive loading:**
1. Render the greeting and StatRow immediately (StatRow shows spinners for its values)
2. Fire all section queries in parallel (`Promise.all` or independent `useEffect` hooks)
3. Each section renders independently as its data arrives
4. Use LoadingSpinner (small variant) as placeholder for each section while loading

This means the user sees the greeting instantly, then sections pop in over ~500ms as queries resolve. Much better than a blank screen for 2 seconds.

```tsx
// Pattern for each section
function ActionsSection() {
  const [actions, setActions] = useState(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    getActiveActions()
      .then(setActions)
      .finally(() => setLoading(false))
  }, [])

  if (loading) return <LoadingSpinner size="sm" />
  if (!actions?.length) return <p className="text-sm text-muted">Nothing due today.</p>
  return (/* render cards */)
}
```

## Navigation from briefing cards

Every tappable card on the briefing navigates to the relevant screen. The simplest approach:

```tsx
import { useRouter } from 'next/navigation'

// In a card's onTap:
router.push('/actions')
router.push('/household')
router.push('/meals')
router.push('/agent-feed')
```

Deep linking (opening a specific item's detail panel on navigation) is nice-to-have. If it's easy to implement (e.g., query param `?id=X` that the target page reads and auto-opens), do it. If not, just navigating to the list is fine for v1.

## Liv's briefing vs. Lee's briefing

The only difference: Liv does not see the Agent Activity section. Everything else is identical.

```tsx
const { isOwner } = useAuth()

return (
  <>
    <Greeting />
    <BriefingStats />
    <ActionsSection />
    {isOwner && <AgentSection />}
    <MealsSection />
    <StaleSection />
    <MaintenanceSection />
  </>
)
```

**Note on actions:** Actions are personal (`user_id` scoped). So Lee's briefing shows Lee's actions and Liv's briefing shows Liv's actions. This is correct — they each see their own to-dos. The household sections (meals, maintenance) show the same shared data for both.
