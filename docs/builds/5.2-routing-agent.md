# 5.2 - Routing Agent

## Metadata

- Date: 2026-04-08
- Spec reference: `docs/specs/5.2-routing-agent.md`

## What Was Built

Routing agent that classifies `pending_routing` actions as human-required or agent-handleable, then dispatches agent-handleable actions to execution agents via the capability registry.

| File | Purpose |
| --- | --- |
| `supabase/migrations/20260408100000_create_agent_capabilities.sql` | Creates `agent_capabilities` table, seeds 5 initial capabilities, adds `routing_scan` task type |
| `supabase/functions/task-runner/index.ts` | Adds `handleRoutingScan()` handler and `dispatchToAgent()` helper |

## Approach

Implemented as a `routing_scan` task type inside the existing `task-runner` edge function, following the same pattern as `triage_scan` (5.1). This gives us watermarking, logging, scheduling, and error handling for free.

### Pipeline flow

```
[Triage Agent] → actions (pending_routing) → [Routing Agent] → human OR [Execution Agent]
```

### Key decisions

- **Batch LLM classification**: All pending actions are sent to the LLM in one call with the full capability list, same as triage batches thoughts.
- **Fallback to human**: If the LLM doesn't return a result for an action, or no capability matches, or the capability is disabled, the action is handed back to human.
- **Dispatch failure handling**: If an execution agent call fails, the action is set to `needs_review` rather than `handed_back`, so the failure is visible.
- **No watermark needed**: Unlike triage (which uses a time watermark to find new thoughts), routing simply queries `agent_status = 'pending_routing'` — the status itself is the marker.
